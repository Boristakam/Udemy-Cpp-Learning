Pointers are variables pointing to an address in memory (meaning that address is their value, and they have a type)
could be address to another var or function 

allows to access data that are outside the scope of a particular function (by using a reference to a pointer param)

Java and Python dont have pointers so they dont allow to directly access memory 
this is by design because they're not made to develop embedded systems, operating systems etc..
Pointers allow complete control over the hardware

important to understand when to use and release memory efficiently 

syntax for declaring a pointers
similar to normal variables ->   (type) (axterix)(pointer name)
e.g.
int *int_ptr       //read from right to left  so it is a pointer to an integer
double *double_ptr //pointer to a double
char *char_ptr     //pointer to a character
string *string_ptr //pointer to a string

they need to be initialised or they will contain garbadge data. in this case an address to anywhere, so you 
could be accessing data at an address you shouldnt be 
e.g.
int *int_ptr {}         //pointer to an integer now points nowhere (address 0) rather than anywhere like in the previous example 
int *int_ptr {nullptr}  //same as above 

All the above are 'typed pointers' because they point to variable of specific type

MAIN USE OF POINTERS: DYNAMIC MEMORY ALLOCATION!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!




Potential pitfalls: (smart pointers take care of most of those problems)
uninitialised pointers 
    they contain garbadge so they can point anywhere and you can change data youre not supposed to. modern day compiler can handle them better
dangling pointers
    pointing to released or invalid addressed (trying to access released data from the heap)
not checking if 'new' failed to allocate memory 
    if new fails exception is thrown and the program stops 
leaking memory 
    forgetting to release memory allocated. if pointer is lost (i.e. when function is popped out of stack), there is no way to access that memory again
    the memory is orphaned or leaked 
    one of the most common pointer problems 
    if you leak enough memory you can run out of memory to allocate 




L-values and R-values:
L-value
    values in memory that have a name and are addressable 
    modifiable if they are not constants 
    i.e x = 100 // x is an l-value
        100 = x // 100 is not an l-value because it is not addressabe (produces a compiler error)
        "frank" = name // "frank" is not an l-value not addressable 

    l-value references
        int x {100};
        int &ref1 = x; // ref1 is a reference to an l-value therefore it is an l-value 
        int &ref1 = 54; // ref1 here is not an l-value` (compiler err.. references are always assigned an address/l-value). same rule applies you pass a value to a function that expects a reference in form of array or pointer or reference 

R-value
    anything that is not an l-value
    on the right hand side of an assignment expression 
    i.e int x = 100 // 100 is an r-value 
            x = y + 100 // (y +100) is an r-value 
            name = "frank" // frank is an r-value
